# AI Tools Usage Report

This report documents how AI tools were used across every phase of AI Arena. It
covers planning, architecture, implementation, testing, and reflection.

## 1. Planning — backlog, user stories, acceptance criteria

**Tools used**

- **Cursor + Claude** for brainstorming the product scope and translating the
  course barem (barem = grading rubric) into user-facing stories.
- **Linear MCP plugin** (this project uses the Linear Cursor plugin) for
  creating and maintaining the backlog. Issues live under the AIA prefix,
  grouped by two milestones: *Prezentare intermediară* and *Prezentare finală*.
- **GitHub issue tracker** for cross-linking bugs discovered during the build.

**How the assistant helped**

- Generated the initial skeleton of user stories (AIA-1 … AIA-17) from a short
  brief of the product idea, then refined each one with explicit acceptance
  criteria.
- Proposed a milestone split: core session flow first (demo-able), analytics +
  polish + docs second.
- Helped translate ambiguous requirements ("should feel live") into concrete
  AC bullets ("vote counts update within 1 second of another user voting via
  Convex real-time subscription").

**Example prompt**

> Take this one-paragraph product idea and return a bulleted list of Linear
> issues. Each issue should be a user story with a clear actor, goal, and at
> least three acceptance criteria written as checkboxes. Group issues under two
> epics: "Session Flow" and "AI Agents". Add labels for `frontend`, `backend`,
> `ai-agent`, `admin`, `real-time` where relevant.

## 2. Architecture — AI-assisted design decisions

**Tools used**

- **Cursor + Claude** for brainstorming data models, discussing trade-offs
  (Convex component vs monolithic schema, AI judge ballots vs weighted
  scoring, anonymized slots vs revealing labels immediately).
- **Mermaid diagrams** generated by the assistant and then hand-edited for
  accuracy — see `docs/ARCHITECTURE.md`.
- **Context7 MCP plugin** for pulling current Convex, TanStack Start, and
  Clerk documentation when evaluating options.

**Design decisions informed by AI**

- Split `sessions`, `rounds`, `roundResponses`, `roundVotes`, `roundAiVotes`,
  `roundArtifacts`, and `sessionEvents` into separate tables so no unbounded
  arrays end up inside a single document (guideline from Convex best
  practices rules the assistant consulted).
- Use anonymized slot letters (`A`, `B`, …) in the client until a round is
  `scored` — keeps voting unbiased.
- Move long-running provider calls into a Node action (`convex/orchestration.ts`
  with `'use node'`) and schedule it from a mutation so the mutation stays
  fast and transactional.
- Count one AI judge ballot per model, explicitly excluding self-votes.

## 3. Implementation — AI-generated vs hand-written code

**Tools used**

- **Cursor agents** (Composer / Claude Sonnet) for generating initial
  scaffolds of components, route files, and Convex functions, then iterating
  over them.
- **shadcn/ui** via the Cursor shadcn skill for consistent primitives
  (tables, cards, dialogs).
- **Playwright / browser MCP** for ad-hoc visual checks during development.

**Breakdown by feature**

| Feature | AI-assisted share | Notes |
|---|---|---|
| `convex/schema.ts` | ~60% | Schema was drafted by the assistant, then manually adjusted to add indexes (`by_session_id_and_access_token_hash`, `by_round_id_and_response_id`) and nullable validators. |
| `convex/orchestration.ts` | ~55% | The provider adapters (OpenAI, Anthropic, Google, Mistral, xAI) and the prompt builders were generated; the timeout wrapper, judging loop, and fallback behaviour were refined by hand. |
| `convex/state.ts` | ~50% | State transitions were sketched by the assistant and then hardened around edge cases (stopped mid-round, aborted rounds, ties). |
| `convex/stats.ts` (new) | ~80% | Cross-session leaderboard, history, and admin cost queries were generated with the current schema as context. |
| `src/routes/*` | ~65% | shadcn scaffolds + Tailwind layout generated; data wiring to Convex and interactive handlers reviewed by a human. |
| `src/components/arena/*` | ~50% | RoundResponseCard, LiveVoteChart and MeasuredEditorialText started from assistant output, heavily tuned for editorial feel. |
| Tests | ~70% | Test skeletons generated; edge cases hand-picked. |
| `docs/ARCHITECTURE.md`, this file | ~85% | Drafted by the assistant and manually verified against current code. |

Hand-written (or heavily rewritten) pieces include most of the editorial copy,
the `pretext` measured text integration, and the exact wording of Critic / Host
prompts.

## 4. Testing — how tests and evals were scaffolded with AI

**Tools used**

- **vitest** with the **`@edge-runtime/vm`** Convex test harness.
- **convex-test** (`t.mutation`, `t.query`, `t.withIdentity`).
- **Cursor agent** to generate test skeletons from a natural description.

**Where AI helped**

- Generated 14+ Convex function tests (`convex/sessions.test.ts`,
  `convex/voting.test.ts`) by reading the schema and related functions and
  proposing scenarios: topic locking, voting restrictions, state transitions,
  invalid round counts.
- Generated pricing / helper tests in `shared/validation.test.ts` with edge
  cases like null token counts and unknown model keys.
- Surfaced scenarios humans had missed (e.g. confirming that
  `sessions.create` rejects a single model, and that
  `stats.listCompletedSessions` returns empty gracefully before any data
  exists).

**Agent evals (lightweight)**

Because the Host and Critic prompts run against live providers, the project
does not ship a synthetic eval harness. Instead we rely on:

- Structural prompt invariants enforced by unit tests (`getThemeCopy` returns
  non-empty `label`, `hostTone`, `criticAngle` for every theme — so Host and
  Critic prompt builders cannot silently produce empty strings).
- Runtime fallbacks: if an agent call fails or demo mode is active, a canned
  `getAgentFallback('host' | 'critic')` message is saved with
  `status: 'fallback'` so the round still completes.

## 5. Reflection — what worked, what did not

**What worked well**

- Letting the assistant generate **Convex schema + mutations** in tandem — the
  schema and the function signatures stayed consistent because both were
  produced from the same context window.
- **Context7 MCP** for up-to-date Convex, TanStack Start, and Clerk docs kept
  the generated code off deprecated paths.
- **Agent-authored tests** caught a number of ordering bugs early.
- **Parallel Cursor agents** to scaffold leaderboard + history + cost tracking
  as three related routes in one pass.

**What required caution**

- The assistant occasionally **reinvented APIs** (e.g. imagined
  `ctx.runAction` from a mutation). Cross-checking with
  `convex/_generated/ai/guidelines.md` was essential.
- Prompt copy that sounds great out of context can leak model identities or
  break the anonymization rule — the human author always needs a final pass.
- Cost estimates are **approximate**. The `MODEL_PRICING` table in
  `shared/arena.ts` is public list pricing, not real invoices; a production
  dashboard would need actual provider billing.

**What we would automate next time**

- Use Cursor hooks to run `pnpm check` on every commit instead of relying on
  the CI workflow alone.
- Add a thin "agent eval" harness that hits mock provider responses and
  asserts on Host/Critic prompt quality (length, presence of theme keywords,
  anonymization respected).
- Add a dedicated `npx convex ai-files install` run to the `pnpm dev` script
  to keep `convex/_generated/ai/guidelines.md` always up to date.
